MAHA_CORE_INTENT_v1_mobile
모바일 열람용 원문 초안

작성 목적:
이 문서는 MAHA 앱의 원래 목적과 개발 방향을 보존하기 위한 기준 문서다.

코드맵은 “현재 코드가 어떻게 생겼는지”를 설명한다.
체인지로그는 “무엇을 언제 바꿨는지”를 설명한다.
이 문서는 “왜 MAHA를 만드는지”를 설명한다.

앞으로 지휘 세션이 마이그레이션되거나,
Provider/API/RAG/Web Search 같은 세부 기능에 몰입해서
방향이 흐려질 때 이 문서를 먼저 참조한다.


==========================
1. MAHA의 한 줄 정의
==========================

MAHA는 단순 채팅 앱이 아니다.

MAHA는 여러 AI 작업자를 안드로이드 앱 안에서
실행하고,
연결하고,
관찰하고,
조정하는
멀티 에이전트 실행 하네스다.

더 구체적으로 말하면:

MAHA는 Provider-native 기능에 의존하지 않고,
앱 내부의 Orchestrator / Worker / Tool / RAG / Memory / File Control 구조로
일반 LLM과 오픈소스 모델의 한계를 보완하는
개인용 AI 실행 시스템이다.


==========================
2. MAHA가 해결하려는 문제
==========================

MAHA가 해결하려는 문제는 단순히 “AI와 대화하기”가 아니다.

진짜 문제는 다음이다.

여러 AI 역할을 나누어 쓰고 싶지만,

- 누가 무엇을 했는지
- 어떤 입력이 어디로 전달됐는지
- 어디서 실패했는지
- 결과가 왜 그렇게 나왔는지

초보자가 직접 관찰하고 제어하기 어렵다.

따라서 MAHA는 다음을 제공해야 한다.

- 역할이 있는 AI 작업자
- 작업자 간 입력·출력 전달
- 순차 실행 흐름
- 중간 결과와 최종 결과 저장
- 실패 지점 표시
- 실행 로그
- 사용자가 이해 가능한 상태 표시
- 나중에 재사용 가능한 시나리오 또는 흐름


==========================
3. 작업모드의 원래 의미
==========================

작업모드는 MAHA의 원형이다.

작업모드의 핵심은 다음이다.

Agent A 실행
→ A의 결과를 Agent B에게 전달
→ B의 결과를 Agent C에게 전달
→ 중간 결과와 최종 결과를 저장
→ 실패 지점과 로그를 확인

작업모드는 다음 목적을 가진다.

- 정해진 Agent들을 순서대로 실행한다.
- 각 Agent의 역할을 명확히 분리한다.
- 이전 결과를 다음 Worker 입력으로 넘긴다.
- 실행 결과, 중간 결과, 실패 지점을 추적한다.
- 사용자가 “작업이 어디까지 진행됐는지” 볼 수 있게 한다.

즉 작업모드는
멀티 에이전트 순차 실행기다.


==========================
4. 대화모드의 원래 의미
==========================

대화모드는 작업모드와 별개의 단순 채팅 기능이 아니다.

대화모드는 작업모드의 철학을
대화형 인터페이스로 확장한 것이다.

흐름은 다음과 같다.

사용자 입력
→ Orchestrator가 요청을 해석
→ 필요한 능력 판단
→ RAG / Memory / Web Search / File / Code Check 사용 여부 결정
→ Main Worker 실행
→ Synthesis Worker가 검증·정리
→ 최종 답변과 실행정보 표시

대화모드의 목적은
“한 모델에게 질문하고 답변 받기”가 아니다.

대화모드의 목적은:

사용자의 자연어 요청을
멀티 에이전트 실행 흐름으로 변환하고,
필요한 도구와 기억과 검색을 연결해,
결과와 과정을 함께 보여주는 것이다.


==========================
5. Orchestrator / Worker 구조의 이유
==========================

대화모드에 Orchestrator와 Worker를 두는 이유는
멀티 에이전트 흉내를 내기 위해서가 아니다.

역할 분리를 위해서다.


--------------------------------------------------
5-1. Orchestrator
--------------------------------------------------

Orchestrator는 사용자의 요청을 해석한다.

담당:

- 요청 의도 파악
- 필요한 capability 판단
- Web Search 필요 여부 판단
- RAG 필요 여부 판단
- Memory 필요 여부 판단
- File 접근 필요 여부 판단
- Code 검증 필요 여부 판단
- 현재 Provider/Model로 가능한지 판단
- 불가능하면 제한 사유를 명시


--------------------------------------------------
5-2. Main Worker
--------------------------------------------------

Main Worker는 핵심 작업을 수행한다.

담당:

- 실제 모델 호출
- 답변 초안 생성
- RAG context 반영
- 검색 결과 반영
- 코드/JSON/테이블 등 출력 형식 준수


--------------------------------------------------
5-3. Synthesis Worker
--------------------------------------------------

Synthesis Worker는 결과를 정리하고 검증한다.

담당:

- 결과 통합
- 형식 보정
- 누락 확인
- 제한 사항 명시
- 최종 답변 생성
- 사용자에게 보여줄 결과 정리


==========================
6. Provider-native와 MAHA-native의 구분
==========================

MAHA는 Provider 기능을 사용할 수 있다.

하지만 Provider 기능에 종속되면 안 된다.


--------------------------------------------------
6-1. Provider-native 기능
--------------------------------------------------

Provider가 자체 제공하는 기능이다.

예:

- Gemini native Web Search
- Groq Compound built-in web/code tools
- OpenAI Responses web_search
- Provider-hosted code execution

이 기능은 가능하면 사용할 수 있다.

하지만 이것은 MAHA의 본체가 아니다.


--------------------------------------------------
6-2. MAHA-native 기능
--------------------------------------------------

MAHA 앱 내부에서 직접 제공해야 하는 기능이다.

예:

- MAHA RAG
- 장기 기억
- 대화 이력 검색
- 로컬 저장소 파일 제어
- 외부 검색 API 연결
- 코드 검증
- Tool Registry
- Worker 간 결과 전달
- 실행 로그
- 실패 지점 표시
- 컨텍스트 제한 극복

MAHA의 본체는 이쪽이다.

핵심 문장:

Provider-native 기능은 보너스다.
MAHA-native harness가 본체다.


==========================
7. Tool calling에 대한 기준
==========================

Tool calling은 단순히
“모델이 도구를 쓸 수 있다”는 뜻으로 보면 안 된다.

반드시 구분해야 한다.

1. 모델이 tool_calls 형식을 출력할 수 있음
2. 앱이 tool_calls를 감지할 수 있음
3. 앱이 해당 tool을 실제 실행할 수 있음
4. 실행 결과를 다시 모델에 넣을 수 있음
5. 최종 답변을 다시 받을 수 있음

현재 MAHA가 우선해야 할 것은 1~2번이다.

3번 이후는 별도 설계가 필요하다.

실제 tool 실행에는 다음이 필요하다.

- Tool Registry
- 허용 tool 목록
- 입력 schema
- 출력 schema
- 사용자 승인 정책
- 실패 처리
- 반복 호출 제한
- 보안 정책
- 로그 기록

따라서 당장 모든 Provider/Model tool을
실행하려고 하면 안 된다.


==========================
8. Capability Layer의 필요성
==========================

MAHA에는 Provider/Model별 기능을 판단하는
중간 계층이 필요하다.

이 계층을 임시로
Capability Layer라고 부른다.

Capability Layer는 다음을 판단한다.

- 사용자 요청이 무엇을 요구하는가?
- 현재 Provider/Model이 native로 가능한가?
- MAHA 내부 tool로 대체 가능한가?
- 둘 다 안 되면 어떻게 제한을 안내할 것인가?

판단 대상:

- 일반 대화
- RAG
- Web Search
- 장기 기억
- 로컬 파일 읽기
- 로컬 파일 쓰기
- 코드 검증
- JSON 출력
- 테이블 출력
- tool_call 감지
- tool 실행
- Provider-native hosted tool


==========================
9. 앞으로의 우선순위
==========================

Provider 추가와 API 연결은 필요하다.

하지만 그것이 MAHA의 목적을 대체하면 안 된다.

앞으로의 우선순위는 다음이어야 한다.

1. MAHA-native Orchestrator 설계
2. Capability Layer 설계
3. Tool Registry 설계
4. RAG 고도화
5. 장기 기억 구조
6. 로컬 저장소 파일 제어
7. 코드 검증 도구
8. 검색 API 연결
9. Provider-native 기능 연동
10. Provider별 고급 기능 최적화

OpenAI Web Search,
Groq Compound,
Gemini native 기능은 중요하지만 후순위다.

그보다 먼저
MAHA가 자체적으로 무엇을 할 수 있어야 하는지
정해야 한다.


==========================
10. 지휘 세션의 판단 기준
==========================

앞으로 모든 작업 지시문을 만들기 전에
다음 질문을 확인한다.

1. 이 작업은 MAHA-native harness를 강화하는가?

2. 아니면 특정 Provider 기능을 따라가는 작업인가?

3. 작업모드의 원래 철학을 훼손하지 않는가?

4. 대화모드가 단순 채팅앱으로 흐르지 않는가?

5. Orchestrator / Worker / Result / Log / Failure Point 구조와 연결되는가?

6. RAG / Memory / Tool / File / Code Check와 장기적으로 연결 가능한가?

7. 사용자가 실행 흐름을 이해하고 제어할 수 있는가?

이 질문에 답하지 못하는 작업은
우선순위를 낮춘다.


==========================
11. 대화모드의 성공 기준 재정의
==========================

대화모드에서 “성공”은
단순히 Provider 호출이 성공했다는 뜻이면 안 된다.

성공 상태는 분리해야 한다.

- Provider 호출: 성공 / 실패
- Worker 실행: 성공 / 실패 / 건너뜀
- 사용자 목표 수행: 성공 / 제한 / 실패
- 제한 사유: 검색 미지원, 도구 미구현, 권한 없음, API Key 없음 등

예를 들어
Groq 일반 모델에게 날씨 검색을 요청했을 때는
다음처럼 표시해야 한다.

Provider 호출: 성공
답변 생성: 성공
사용자 목표 수행: 제한됨
제한 사유:
현재 Provider/Model에서는 MAHA Web Search tool이 연결되지 않음

이렇게 해야 사용자가
“앱 호출은 성공했지만,
내가 원한 작업은 제한되었구나”
라고 이해할 수 있다.


==========================
12. 마이그레이션 시 반드시 함께 넘길 압축본
==========================

MAHA는 단순 채팅앱이나 Provider 관리앱이 아니다.

MAHA는 여러 AI 작업자를 안드로이드에서
실행하고,
연결하고,
관찰하고,
조정하는
멀티 에이전트 실행 하네스다.

작업모드는 이 철학의 원형이다.

Agent들을 순차 실행하고,
이전 결과를 다음 Agent 입력으로 전달하며,
중간 결과·최종 결과·로그·실패 지점을
저장하고 보여준다.

대화모드는 이 철학을 대화형으로 확장한 것이다.

사용자 입력을 Orchestrator가 해석하고,
필요한 capability를 판단하며,
Main Worker와 Synthesis Worker가 실행·검증·정리한다.

Provider-native 기능은 보너스다.

Gemini native Web Search,
Groq Compound,
OpenAI Responses Web Search 같은 기능은 활용할 수 있지만
MAHA의 본체가 아니다.

MAHA의 본체는 MAHA-native harness다.

RAG,
장기 기억,
대화 이력 검색,
로컬 파일 제어,
검색 API,
코드 검증,
Tool Registry,
Worker 간 결과 전달,
실행 로그,
실패 지점 표시를
앱 내부 구조로 제공해야 한다.

앞으로 모든 작업은 다음 질문으로 판단한다.

이 작업이 MAHA-native harness를 강화하는가?
대화모드가 단순 채팅앱으로 흐르지 않는가?
Orchestrator / Worker / Result / Log / Failure Point 구조와 연결되는가?
사용자가 실행 흐름을 이해하고 제어할 수 있는가?


==========================
13. 현재 개발 방향 보정
==========================

현재까지의 Provider/API/RAG/Web Search 작업은
버릴 것이 아니다.

다만 위치를 다시 잡아야 한다.

- Provider 관리는 하네스의 실행 엔진 선택 기능이다.
- Model 관리는 Worker별 도구 선택 기능이다.
- RAG는 MAHA-native 기억/검색의 시작점이다.
- Web Search는 Provider-native와 MAHA-native 두 축으로 분리한다.
- 실행정보 UI는 사용자가 하네스를 관찰하는 계기판이다.
- 코드/JSON/테이블 렌더링은 Worker 결과를 구조화해 보여주는 표시 장치다.
- Tool call 감지는 Tool Registry로 가기 전의 진단 단계다.

즉 최근 구현은 모두 의미가 있다.

하지만 앞으로는
“Provider 기능 확장”이 아니라
“MAHA 하네스 강화”라는 관점으로 재배치해야 한다.


==========================
14. 초기 설계 문서에서 복원한 원래 의도
==========================

초기 설계에서 MAHA는 다음처럼 정의되었다.

- 에이전트 = 역할이 있는 AI 작업자
- 하네스 = 여러 작업자를 묶어서 움직이게 하는 제어 장치

MAHA는 다음을 관리하는 앱이다.

- 어떤 에이전트가
- 어떤 순서로
- 어떤 입력을 받아
- 어떤 결과를 내는지

이것이 MAHA의 첫 목적이었다.

초기 설계의 핵심 기능은 다음이었다.

1. 에이전트 목록 관리
2. 에이전트 상세 설정
3. 단일 에이전트 실행
4. 멀티 에이전트 순차 실행
5. 실행 흐름 화면
6. 입력값 전달 기능
7. 결과 기록 보기
8. 실행 로그 보기
9. 템플릿 시나리오 저장
10. 실패 지점 표시

이 10개는 작업모드뿐 아니라
대화모드에도 철학적으로 이어져야 한다.


==========================
15. 대화모드에 초기 설계를 적용하는 방법
==========================

초기 작업모드의 Agent 개념은
대화모드에서 다음처럼 확장된다.

작업모드 Agent:
- Planner
- Researcher
- Writer
- Reviewer

대화모드 Worker:
- Orchestrator
- Main Worker
- Synthesis Worker
- Tool Worker
- Memory Worker
- RAG Worker
- Code Check Worker

작업모드는 사용자가 정한 순서를 실행한다.

대화모드는 사용자 자연어 요청을 받아
Orchestrator가 필요한 순서를 계획한다.

즉 대화모드는
“자동 시나리오 생성형 작업모드”에 가깝다.


==========================
16. Provider와 Model의 위치
==========================

Provider와 Model은 MAHA의 목적 그 자체가 아니다.

Provider와 Model은
Worker가 작업을 수행하기 위해 사용하는 실행 엔진이다.

따라서 Provider 관리와 Model 관리는 중요하지만,
그 자체가 MAHA의 중심 기능은 아니다.

정확한 위치는 다음이다.

Provider 관리:
- 어떤 외부/로컬 추론 엔진을 사용할지 선택하는 기능

Model 관리:
- 어떤 Worker가 어떤 능력의 모델을 쓸지 선택하는 기능

Capability:
- 그 모델이 어떤 형식의 작업에 적합한지 판단하는 기능

Tool:
- 모델이 못 하는 일을 MAHA가 대신 수행하는 기능

RAG/Memory:
- 모델의 기억 부족과 컨텍스트 제한을 보완하는 기능


==========================
17. 앞으로 지켜야 할 금지 방향
==========================

다음 방향으로 흐르면 MAHA의 원래 목적이 약해진다.

- Provider 설정 앱으로 변하는 것
- 단순 채팅 앱으로 변하는 것
- 특정 Provider의 고급 기능을 따라가는 데 매몰되는 것
- Worker 구조가 이름만 남고 실제 역할 분리가 사라지는 것
- 실행정보가 단순 디버그 로그로만 남는 것
- 실패 지점이 사용자에게 명확히 보이지 않는 것
- RAG/Memory/File/Tool이 부가기능처럼 밀려나는 것

앞으로는 기능을 추가할 때마다
이 금지 방향에 해당하지 않는지 확인해야 한다.


==========================
18. 앞으로의 개발 기준 문장
==========================

이 문장을 기준으로 삼는다.

MAHA는 모델을 많이 붙이는 앱이 아니다.
MAHA는 모델을 작업자로 조직하는 앱이다.

MAHA는 Provider 기능을 모으는 앱이 아니다.
MAHA는 Provider의 한계를 내부 하네스로 보완하는 앱이다.

MAHA는 단순히 답변을 받는 앱이 아니다.
MAHA는 요청이 어떻게 처리되었는지 사용자가 관찰하고 조정하는 앱이다.

MAHA는 기억 없는 채팅창이 아니다.
MAHA는 사용자의 작업 맥락과 기록을 쌓아가는 개인 AI 실행 시스템이다.
