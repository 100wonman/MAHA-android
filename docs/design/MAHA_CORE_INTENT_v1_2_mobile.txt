MAHA_CORE_INTENT_v1.2_mobile
모바일 열람용 기준 문서

버전: v1.2

작성 목적:
이 문서는 MAHA 앱의 원래 목적과 개발 방향을 보존하기 위한 기준 문서다.

코드맵은 “현재 코드가 어떻게 생겼는지”를 설명한다.
체인지로그는 “무엇을 언제 바꿨는지”를 설명한다.
이 문서는 “왜 MAHA를 만드는지”를 설명한다.

앞으로 지휘 세션이 마이그레이션되거나,
Provider/API/RAG/Web Search 같은 세부 기능에 몰입해서
방향이 흐려질 때 이 문서를 먼저 참조한다.


==================================================
0. v1.2 업데이트 핵심
==================================================

v1.1 핵심:
초기 작업모드의 순차 실행은 V1/rate limit/free API 제약에 따른 안전한 시작 구조였다.
MAHA의 장기 실행 철학은 반드시 순차 실행만이 아니다.

대화모드와 향후 작업모드는 Orchestrator가 작업 의존성을 판단해
단일 실행, 순차 실행, 병렬 실행, 혼합 실행을 선택하는 구조로 확장되어야 한다.

v1.2 추가 핵심:
MAHA의 Worker 구조는 고정된 개체 수를 전제로 하지 않는다.

Orchestrator, Main Worker, Synthesis Worker, RAG Worker, Tool Worker, Memory Worker,
Code Check Worker 같은 이름은 고정된 앱 내부 개체가 아니라 기본 역할 템플릿이다.

실제 Worker 구성은 사용자가 추가, 삭제, 수정할 수 있어야 한다.

각 Worker는 다음을 사용자 설정값으로 가져야 한다.

- Worker 이름
- 역할 설명
- System Instruction
- Provider
- Model
- Capability override
- 실행 순서
- 병렬 실행 가능 여부
- 입력 정책
- 출력 정책
- 활성/비활성 상태

즉, MAHA는 고정된 3-worker 채팅 구조가 아니다.

MAHA는 사용자가 LLM 작업자들을 구성하고 조직하는
가변 멀티 에이전트 하네스다.


==================================================
1. MAHA의 한 줄 정의
==================================================

MAHA는 단순 채팅 앱이 아니다.

MAHA는 여러 AI 작업자를 안드로이드 앱 안에서
실행하고, 연결하고, 관찰하고, 조정하는
멀티 에이전트 실행 하네스다.

더 구체적으로 말하면:

MAHA는 Provider-native 기능에 의존하지 않고,
앱 내부의 Orchestrator / Worker / Tool / RAG / Memory / File Control 구조로
일반 LLM과 오픈소스 모델의 한계를 보완하는 개인용 AI 실행 시스템이다.

v1.2 기준으로 더 정확히 말하면:

MAHA는 고정된 Worker 몇 개를 제공하는 앱이 아니다.

MAHA는 사용자가 여러 LLM Worker를 만들고,
각 Worker의 역할과 System Instruction과 Provider/Model을 정하고,
그 Worker들을 단일/순차/병렬/혼합 실행으로 조직하는
개인용 멀티 에이전트 하네스다.


==================================================
2. MAHA가 해결하려는 문제
==================================================

MAHA가 해결하려는 문제는 단순히 “AI와 대화하기”가 아니다.

진짜 문제는 다음이다.

여러 AI 역할을 나누어 쓰고 싶지만,

- 누가 무엇을 했는지
- 어떤 입력이 어디로 전달됐는지
- 어디서 실패했는지
- 결과가 왜 그렇게 나왔는지
- 어떤 Worker가 어떤 역할과 지시문으로 동작했는지
- 어떤 Provider/Model이 어떤 Worker에 배정되었는지

초보자가 직접 관찰하고 제어하기 어렵다.

따라서 MAHA는 다음을 제공해야 한다.

- 역할이 있는 AI 작업자
- 사용자가 편집 가능한 Worker Profile
- 사용자가 편집 가능한 System Instruction
- Worker별 Provider/Model 선택
- Worker 간 입력·출력 전달
- 단일 실행 흐름
- 순차 실행 흐름
- 병렬 실행 흐름
- 순차/병렬이 섞인 혼합 실행 흐름
- 중간 결과와 최종 결과 저장
- 실패 지점 표시
- 실행 로그
- 사용자가 이해 가능한 상태 표시
- 나중에 재사용 가능한 시나리오 또는 흐름


==================================================
3. 작업모드의 원래 의미
==================================================

작업모드는 MAHA의 원형이다.

초기 작업모드의 핵심은 다음이었다.

Agent A 실행
→ A의 결과를 Agent B에게 전달
→ B의 결과를 Agent C에게 전달
→ 중간 결과와 최종 결과를 저장
→ 실패 지점과 로그를 확인

초기 V1에서 순차 실행을 먼저 선택한 이유는
MAHA의 최종 철학이 순차 실행뿐이기 때문이 아니었다.

당시에는 다음 제약이 있었다.

- 무료 API 사용량 제한
- rate limit 문제
- Provider 수 부족
- 안정적인 첫 구조 필요
- 초보자도 이해 가능한 실행 흐름 필요
- 작은 단위로 앱을 성공시키는 것이 우선

따라서 초기 작업모드의 순차 실행은 안전한 출발점이었다.

하지만 MAHA의 최종 실행 방식은 다음으로 확장되어야 한다.

- 단일 실행
- 순차 실행
- 병렬 실행
- 혼합 실행


--------------------------------------------------
3-1. 순차 실행
--------------------------------------------------

순차 실행은 이전 Worker 결과가 다음 Worker 입력으로 필요한 경우에 사용한다.

예:

Planner
→ Researcher
→ Writer
→ Reviewer

또는:

요청 분석
→ 자료 검색
→ 초안 작성
→ 검증
→ 최종 정리

순차 실행은 다음 상황에 적합하다.

- 앞 단계 결과가 다음 단계의 입력이 되는 경우
- 추론 흐름이 단계적으로 쌓이는 경우
- 실패 지점을 명확히 추적해야 하는 경우
- 안정성이 속도보다 중요한 경우


--------------------------------------------------
3-2. 병렬 실행
--------------------------------------------------

병렬 실행은 서로 독립적인 하위 작업을 동시에 처리할 수 있는 경우에 사용한다.

예:

- 여러 관점에서 같은 주제 분석
- 여러 검색 소스 동시 조회
- 여러 모델에 같은 질문을 보내 비교
- 코드의 문법 검토와 로직 검토를 분리 수행
- 긴 문서를 여러 chunk로 나누어 요약
- 여러 Provider의 응답을 비교

병렬 실행은 다음 상황에 적합하다.

- 하위 작업들이 서로의 결과를 기다릴 필요가 없는 경우
- 속도가 중요한 경우
- 여러 모델/Provider를 비교하고 싶은 경우
- 검색, 요약, 분류, 검토를 나눠서 동시에 수행할 수 있는 경우


--------------------------------------------------
3-3. 혼합 실행
--------------------------------------------------

혼합 실행은 순차 실행과 병렬 실행을 함께 쓰는 방식이다.

예:

사용자 요청
→ Orchestrator가 작업 계획 수립
→ Research Worker 1, 2, 3 병렬 실행
→ Code Check Worker 병렬 실행
→ RAG Worker 병렬 실행
→ Synthesis Worker가 결과 병합
→ Reviewer Worker가 최종 검증

혼합 실행은 MAHA의 장기 목표에 가장 가깝다.

핵심은 다음이다.

Orchestrator가 작업 의존성을 판단한다.
독립 작업은 병렬로 실행한다.
의존 작업은 순차로 실행한다.
마지막에 Synthesis Worker가 통합한다.


==================================================
4. 대화모드의 원래 의미
==================================================

대화모드는 작업모드와 별개의 단순 채팅 기능이 아니다.

대화모드는 작업모드의 철학을 대화형 인터페이스로 확장한 것이다.

흐름은 다음과 같다.

사용자 입력
→ Orchestrator가 요청을 해석
→ 필요한 능력 판단
→ 작업 의존성 판단
→ 사용 가능한 Worker Profile 확인
→ 순차/병렬/혼합 실행 계획 생성
→ RAG / Memory / Web Search / File / Code Check 사용 여부 결정
→ Main Worker 또는 여러 하위 Worker 실행
→ Synthesis Worker가 검증·정리
→ 최종 답변과 실행정보 표시

대화모드의 목적은 “한 모델에게 질문하고 답변 받기”가 아니다.

대화모드의 목적은:

사용자의 자연어 요청을 멀티 에이전트 실행 흐름으로 변환하고,
필요한 도구와 기억과 검색을 연결해,
결과와 과정을 함께 보여주는 것이다.

대화모드는 장기적으로 “자동 시나리오 생성형 작업모드”에 가깝다.


==================================================
5. 가변 Worker 구조 원칙
==================================================

MAHA의 Worker 구조는 고정 개체 수를 전제로 하지 않는다.

Orchestrator, Main Worker, Synthesis Worker, RAG Worker, Memory Worker, Tool Worker,
Web Search Worker, Code Check Worker 같은 이름은
앱이 영구적으로 고정해야 하는 개체명이 아니다.

이 이름들은 기본 역할 템플릿이다.

사용자는 실제 대화모드 또는 시나리오에서
필요한 Worker를 추가하고,
삭제하고,
수정하고,
비활성화할 수 있어야 한다.

즉:

고정된 3개 Worker 구조가 아니다.
고정된 Orchestrator 1개 + Main 1개 + Synthesis 1개만 쓰는 앱도 아니다.

MAHA는 사용자가 LLM 작업자를 조직하는 앱이다.


--------------------------------------------------
5-1. Worker Profile
--------------------------------------------------

각 Worker는 장기적으로 Worker Profile로 관리되어야 한다.

Worker Profile 후보 필드:

- workerProfileId
- displayName
- roleLabel
- roleDescription
- systemInstruction
- providerId
- modelId
- capabilityOverrides
- inputPolicy
- outputPolicy
- executionOrder
- canRunInParallel
- dependsOnWorkerIds
- enabled
- createdAt
- updatedAt

핵심:

System Instruction은 반드시 사용자가 수정 가능해야 한다.

Provider와 Model도 Worker별로 선택 가능해야 한다.

Worker별 capability override도 사용자가 조정할 수 있어야 한다.


--------------------------------------------------
5-2. System Instruction 편집 가능 원칙
--------------------------------------------------

MAHA에서 System Instruction은 앱 내부에 고정된 문구로 박혀 있으면 안 된다.

기본 템플릿은 제공할 수 있다.

예:

- 기본 Orchestrator instruction
- 기본 Researcher instruction
- 기본 Writer instruction
- 기본 Reviewer instruction
- 기본 Synthesis instruction
- 기본 Code Check instruction

하지만 이것은 시작점일 뿐이다.

사용자는 각 Worker의 System Instruction을 수정할 수 있어야 한다.

이것은 “파인튜닝” 자체는 아니지만,
사용자 입장에서 Worker의 행동을 조정하는 가장 중요한 방법이다.


--------------------------------------------------
5-3. 기본 템플릿과 사용자 설정의 관계
--------------------------------------------------

앱은 기본 템플릿을 제공한다.

예:

- Orchestrator
- Main Worker
- Synthesis Worker
- RAG Worker
- Memory Worker
- Tool Worker
- Web Search Worker
- Code Check Worker
- Comparison Worker
- Reviewer Worker

하지만 실제 실행은 사용자 설정을 우선한다.

기본 템플릿:
초기 사용자를 돕기 위한 시작점

사용자 설정:
실제 앱이 따라야 하는 기준

따라서 앱 내부 구현은
“이 Worker는 항상 이런 instruction을 가진다”
라고 고정하지 않아야 한다.


--------------------------------------------------
5-4. Conversation Scenario / Worker Set
--------------------------------------------------

여러 Worker Profile은 하나의 Scenario 또는 Worker Set으로 묶일 수 있다.

후보 개념:

ConversationScenarioProfile

후보 필드:

- scenarioId
- name
- description
- workerProfileIds
- defaultExecutionMode
- orchestratorProfileId
- synthesisProfileId
- userEditable
- createdAt
- updatedAt

목적:

- 사용자가 자주 쓰는 Worker 구성을 저장
- 대화 목적별로 다른 Worker 조합 사용
- 예: 코딩 검토 세트, 조사/요약 세트, 글쓰기 세트, 개인비서 세트

이 구조는 작업모드의 “템플릿 시나리오 저장” 철학을 대화모드로 확장한 것이다.


==================================================
6. Orchestrator / Worker 구조의 이유
==================================================

대화모드에 Orchestrator와 Worker를 두는 이유는
멀티 에이전트 흉내를 내기 위해서가 아니다.

역할 분리를 위해서다.

다만 v1.2 기준으로,
Orchestrator와 Worker는 고정 개체가 아니라
사용자가 구성 가능한 Worker Profile의 한 종류로 봐야 한다.


--------------------------------------------------
6-1. Orchestrator
--------------------------------------------------

Orchestrator는 사용자의 요청을 해석한다.

담당:

- 요청 의도 파악
- 필요한 capability 판단
- Web Search 필요 여부 판단
- RAG 필요 여부 판단
- Memory 필요 여부 판단
- File 접근 필요 여부 판단
- Code 검증 필요 여부 판단
- 현재 Provider/Model로 가능한지 판단
- 사용 가능한 Worker Profile 확인
- 작업 의존성 판단
- 순차 실행이 필요한지 판단
- 병렬 실행이 가능한지 판단
- 혼합 실행 계획 생성
- 불가능하면 제한 사유를 명시

Orchestrator의 핵심 질문:

이 요청은 한 번의 모델 호출로 충분한가?
여러 Worker로 나눠야 하는가?
각 Worker가 서로 독립적인가?
어떤 Worker 결과가 다른 Worker 입력으로 필요한가?
현재 사용자가 설정한 Worker Profile로 처리 가능한가?
어떤 기능이 Provider-native이고,
어떤 기능이 MAHA-native로 보완되어야 하는가?


--------------------------------------------------
6-2. Main Worker
--------------------------------------------------

Main Worker는 핵심 작업을 수행한다.

담당:

- 실제 모델 호출
- 답변 초안 생성
- RAG context 반영
- 검색 결과 반영
- 코드/JSON/테이블 등 출력 형식 준수

주의:
Main Worker도 고정 1개일 필요는 없다.

요청이나 Scenario에 따라
여러 Main Worker 또는 Specialized Worker가 있을 수 있다.


--------------------------------------------------
6-3. Parallel Workers
--------------------------------------------------

Parallel Worker는 서로 독립적인 하위 작업을 수행한다.

예:

- Search Worker
- RAG Worker
- Memory Worker
- Code Check Worker
- Comparison Worker
- Summary Worker
- Fact Check Worker
- Format Worker

역할:

- 독립적인 작업을 병렬로 수행
- 결과를 Synthesis Worker에게 전달
- 실패 시 해당 Worker의 실패 지점과 사유를 남김


--------------------------------------------------
6-4. Synthesis Worker
--------------------------------------------------

Synthesis Worker는 결과를 정리하고 검증한다.

담당:

- 여러 Worker 결과 통합
- 형식 보정
- 누락 확인
- 충돌 결과 비교
- 제한 사항 명시
- 최종 답변 생성
- 사용자에게 보여줄 결과 정리

주의:
Synthesis Worker도 고정 문구로 박으면 안 된다.
사용자가 Synthesis 방식과 System Instruction을 편집할 수 있어야 한다.


==================================================
7. Provider-native와 MAHA-native의 구분
==================================================

MAHA는 Provider 기능을 사용할 수 있다.

하지만 Provider 기능에 종속되면 안 된다.

Provider-native 기능:
- Gemini native Web Search
- Groq Compound built-in web/code tools
- OpenAI Responses web_search
- Provider-hosted code execution

MAHA-native 기능:
- MAHA RAG
- 장기 기억
- 대화 이력 검색
- 로컬 저장소 파일 제어
- 외부 검색 API 연결
- 코드 검증
- Tool Registry
- Worker 간 결과 전달
- 실행 로그
- 실패 지점 표시
- 컨텍스트 제한 극복
- 순차/병렬/혼합 실행 계획
- 사용자 편집 가능한 Worker Profile
- 사용자 편집 가능한 System Instruction

핵심 문장:

Provider-native 기능은 보너스다.
MAHA-native harness가 본체다.


==================================================
8. Tool calling에 대한 기준
==================================================

Tool calling은 단순히
“모델이 도구를 쓸 수 있다”는 뜻으로 보면 안 된다.

반드시 구분해야 한다.

1. 모델이 tool_calls 형식을 출력할 수 있음
2. 앱이 tool_calls를 감지할 수 있음
3. 앱이 해당 tool을 실제 실행할 수 있음
4. 실행 결과를 다시 모델에 넣을 수 있음
5. 최종 답변을 다시 받을 수 있음

현재 MAHA가 우선해야 할 것은 1~2번이다.

3번 이후는 별도 설계가 필요하다.

실제 tool 실행에는 다음이 필요하다.

- Tool Registry
- 허용 tool 목록
- 입력 schema
- 출력 schema
- 사용자 승인 정책
- 실패 처리
- 반복 호출 제한
- 보안 정책
- 로그 기록

따라서 당장 모든 Provider/Model tool을 실행하려고 하면 안 된다.


==================================================
9. Capability Layer의 필요성
==================================================

MAHA에는 Provider/Model별 기능과 Worker별 설정을 판단하는 중간 계층이 필요하다.

이 계층을 임시로 Capability Layer라고 부른다.

Capability Layer는 다음을 판단한다.

- 사용자 요청이 무엇을 요구하는가?
- 현재 Provider/Model이 native로 가능한가?
- 현재 Worker Profile 설정으로 가능한가?
- 사용자가 특정 Worker capability를 override했는가?
- MAHA 내부 tool로 대체 가능한가?
- 이 작업은 순차로 해야 하는가?
- 병렬로 나눌 수 있는가?
- 순차/병렬 혼합이 필요한가?
- 둘 다 안 되면 어떻게 제한을 안내할 것인가?

판단 대상:

- 일반 대화
- RAG
- Web Search
- 장기 기억
- 로컬 파일 읽기
- 로컬 파일 쓰기
- 코드 검증
- JSON 출력
- 테이블 출력
- tool_call 감지
- tool 실행
- Provider-native hosted tool
- 순차 실행
- 병렬 실행
- 혼합 실행
- Worker Profile
- System Instruction
- Scenario Profile

14. Provider별 고급 기능 최적화


==================================================
11. 지휘 세션의 판단 기준
==================================================

앞으로 모든 작업 지시문을 만들기 전에 다음 질문을 확인한다.

1. 이 작업은 MAHA-native harness를 강화하는가?
2. 아니면 특정 Provider 기능을 따라가는 작업인가?
3. 작업모드의 원래 철학을 훼손하지 않는가?
4. 대화모드가 단순 채팅앱으로 흐르지 않는가?
5. Orchestrator / Worker / Result / Log / Failure Point 구조와 연결되는가?
6. RAG / Memory / Tool / File / Code Check와 장기적으로 연결 가능한가?
7. 사용자가 실행 흐름을 이해하고 제어할 수 있는가?
8. 이 작업은 순차/병렬/혼합 실행 철학과 충돌하지 않는가?
9. 이 기능이 Provider-native 기능인지, MAHA-native harness 기능인지 명확한가?
10. Worker 개수와 역할을 정적으로 고정하지 않는가?
11. System Instruction을 사용자가 수정할 수 있는 구조를 보존하는가?
12. Worker별 Provider/Model 선택 가능성을 닫아버리지 않는가?

이 질문에 답하지 못하는 작업은 우선순위를 낮춘다.


==================================================
12. 대화모드의 성공 기준 재정의
==================================================

대화모드에서 “성공”은
단순히 Provider 호출이 성공했다는 뜻이면 안 된다.

성공 상태는 분리해야 한다.

- Provider 호출: 성공 / 실패
- Worker 실행: 성공 / 실패 / 건너뜀
- 사용자 목표 수행: 성공 / 제한 / 실패
- 실행 방식: 단일 / 순차 / 병렬 / 혼합
- 제한 사유: 검색 미지원, 도구 미구현, 권한 없음, API Key 없음 등

예:

Groq 일반 모델에게 날씨 검색을 요청했을 때:

Provider 호출: 성공
답변 생성: 성공
사용자 목표 수행: 제한됨
제한 사유:
현재 Provider/Model에서는 MAHA Web Search tool이 연결되지 않음


==================================================
13. 현재 개발 방향 보정
==================================================

현재까지의 Provider/API/RAG/Web Search 작업은 버릴 것이 아니다.

다만 위치를 다시 잡아야 한다.

- Provider 관리는 하네스의 실행 엔진 선택 기능이다.
- Model 관리는 Worker별 도구 선택 기능이다.
- Worker Profile은 사용자 구성 가능한 작업자 정의다.
- System Instruction은 Worker별 사용자 편집 가능 행동 정책이다.
- RAG는 MAHA-native 기억/검색의 시작점이다.
- Web Search는 Provider-native와 MAHA-native 두 축으로 분리한다.
- 실행정보 UI는 사용자가 하네스를 관찰하는 계기판이다.
- 코드/JSON/테이블 렌더링은 Worker 결과를 구조화해 보여주는 표시 장치다.
- Tool call 감지는 Tool Registry로 가기 전의 진단 단계다.
- 순차/병렬/혼합 실행은 Orchestrator의 핵심 판단 대상이다.


==================================================
14. 초기 설계 문서에서 복원한 원래 의도
==================================================

초기 설계에서 MAHA는 다음처럼 정의되었다.

- 에이전트 = 역할이 있는 AI 작업자
- 하네스 = 여러 작업자를 묶어서 움직이게 하는 제어 장치

MAHA는 다음을 관리하는 앱이다.

- 어떤 에이전트가
- 어떤 순서로
- 어떤 입력을 받아
- 어떤 결과를 내는지

이것이 MAHA의 첫 목적이었다.

초기 설계의 핵심 기능은 다음이었다.

1. 에이전트 목록 관리
2. 에이전트 상세 설정
3. 단일 에이전트 실행
4. 멀티 에이전트 순차 실행
5. 실행 흐름 화면
6. 입력값 전달 기능
7. 결과 기록 보기
8. 실행 로그 보기
9. 템플릿 시나리오 저장
10. 실패 지점 표시

v1.2 기준으로,
이 “에이전트”는 고정 역할 개체가 아니라
사용자가 구성 가능한 Worker Profile로 확장되어야 한다.


==================================================
15. 대화모드에 초기 설계를 적용하는 방법
==================================================

초기 작업모드의 Agent 개념은 대화모드에서 다음처럼 확장된다.

작업모드 Agent:
- Planner
- Researcher
- Writer
- Reviewer

대화모드 Worker Profile:
- 사용자가 이름을 정함
- 사용자가 역할 설명을 정함
- 사용자가 System Instruction을 정함
- 사용자가 Provider/Model을 정함
- 사용자가 capability override를 정함
- 사용자가 실행 순서와 병렬 가능 여부를 정함

기본 템플릿 예:

- Orchestrator
- Main Worker
- Synthesis Worker
- RAG Worker
- Memory Worker
- Tool Worker
- Web Search Worker
- Code Check Worker
- Comparison Worker
- Reviewer Worker

작업모드는 사용자가 정한 순서를 실행한다.

대화모드는 사용자 자연어 요청을 받아
Orchestrator가 필요한 실행 계획을 세운다.

그 계획은 다음 중 하나일 수 있다.

- 단일 실행
- 순차 실행
- 병렬 실행
- 혼합 실행

즉 대화모드는
“자동 시나리오 생성형 작업모드”에 가깝다.


==================================================
16. Provider와 Model의 위치
==================================================

Provider와 Model은 MAHA의 목적 그 자체가 아니다.

Provider와 Model은 Worker가 작업을 수행하기 위해 사용하는 실행 엔진이다.

정확한 위치는 다음이다.

Provider 관리:
- 어떤 외부/로컬 추론 엔진을 사용할지 선택하는 기능

Model 관리:
- 어떤 Worker가 어떤 능력의 모델을 쓸지 선택하는 기능

Worker Profile:
- 어떤 역할의 작업자가 어떤 지시문과 모델로 동작할지 정의하는 기능

System Instruction:
- Worker의 행동 방식과 판단 기준을 정하는 사용자 편집 가능 정책

Capability:
- 그 모델과 Worker가 어떤 형식의 작업에 적합한지 판단하는 기능

Tool:
- 모델이 못 하는 일을 MAHA가 대신 수행하는 기능

RAG/Memory:
- 모델의 기억 부족과 컨텍스트 제한을 보완하는 기능

Execution Planner:
- 작업을 단일/순차/병렬/혼합 중 어떤 방식으로 실행할지 결정하는 기능


==================================================
17. 앞으로 지켜야 할 금지 방향
==================================================

다음 방향으로 흐르면 MAHA의 원래 목적이 약해진다.

- Provider 설정 앱으로 변하는 것
- 단순 채팅 앱으로 변하는 것
- 특정 Provider의 고급 기능을 따라가는 데 매몰되는 것
- Worker 구조가 이름만 남고 실제 역할 분리가 사라지는 것
- Worker 개수와 역할이 앱 내부에 정적으로 고정되는 것
- Worker System Instruction이 사용자가 수정할 수 없는 고정값이 되는 것
- Worker별 Provider/Model 선택 가능성이 닫히는 것
- 실행정보가 단순 디버그 로그로만 남는 것
- 실패 지점이 사용자에게 명확히 보이지 않는 것
- RAG/Memory/File/Tool이 부가기능처럼 밀려나는 것
- 순차 실행만을 MAHA의 영구 구조로 오해하는 것
- 병렬 실행 가능성을 구조적으로 닫아버리는 것

18. 앞으로의 개발 기준 문장
==================================================

MAHA는 모델을 많이 붙이는 앱이 아니다.
MAHA는 모델을 작업자로 조직하는 앱이다.

MAHA는 Provider 기능을 모으는 앱이 아니다.
MAHA는 Provider의 한계를 내부 하네스로 보완하는 앱이다.

MAHA는 단순히 답변을 받는 앱이 아니다.
MAHA는 요청이 어떻게 처리되었는지 사용자가 관찰하고 조정하는 앱이다.

MAHA는 기억 없는 채팅창이 아니다.
MAHA는 사용자의 작업 맥락과 기록을 쌓아가는 개인 AI 실행 시스템이다.

MAHA는 순차 실행만 하는 앱이 아니다.
MAHA는 Orchestrator가 작업 의존성을 판단해
단일/순차/병렬/혼합 실행을 선택하는 앱이다.

MAHA는 고정 Worker 몇 개를 쓰는 앱이 아니다.
MAHA는 사용자가 Worker와 System Instruction과 Provider/Model을 구성하는 앱이다.

20. 작업세션 전달용 핵심 압축문
==================================================

MAHA는 단순 채팅앱이 아니라
MAHA-native multi-agent harness다.

초기 작업모드는 순차 실행 중심이었지만,
이는 V1/rate limit/free API 제약에 따른 안전한 시작 구조였다.

대화모드는 Orchestrator가 작업 의존성을 판단해
단일/순차/병렬/혼합 실행을 선택하는 구조로 확장한다.

MAHA의 Worker 구조는 고정 개체 수를 전제로 하지 않는다.

Orchestrator, Main Worker, Synthesis Worker 같은 이름은
기본 역할 템플릿이다.

실제 Worker 구성은 사용자가 추가, 삭제, 수정할 수 있어야 한다.

각 Worker는 사용자가 편집 가능한
System Instruction,
Provider,
Model,
Capability override,
실행 순서,
병렬 실행 가능 여부,
입력 정책,
출력 정책을 가져야 한다.

Provider-native 기능은 보너스이며,
MAHA-native Orchestrator / Worker / RAG / Memory / Tool / File / Code Check 구조가 본체다.

Provider와 Model은 목적이 아니라 Worker의 실행 엔진이다.

앞으로 작업은
Provider 기능 확장이 아니라
MAHA 하네스 강화 관점에서 판단한다.


==================================================
21. 마지막 확인 문장
==================================================

앞으로 지휘 세션은
이 문서를 코드맵, 체인지로그와 동급의 기준 자료로 삼는다.

코드맵은 구조를 알려준다.
체인지로그는 이력을 알려준다.
이 문서는 방향을 알려준다.

방향이 흔들릴 때는
새 기능을 추가하기 전에
이 문서를 먼저 확인한다.
